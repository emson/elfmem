## SQLAlchemy ORM vs Core

SQLAlchemy has two distinct APIs:

```
ORM:   Maps Python classes to tables. Objects have attributes that lazy-load
       related rows. Session manages identity and change tracking.

Core:  SQL Expression Language. Builds and executes SQL explicitly.
       Returns rows as dicts/tuples. No lazy loading, no magic.
```

### Why ORM is wrong for AMGS

**1. The N+1 centrality problem**

Centrality requires summing edge weights per block. With ORM:

```python
for block in session.query(Block).filter_by(status="active").all():
    centrality = sum(e.weight for e in block.edges)  # ← 1 query per block
```

This fires one SELECT per block — 50 queries for 50 blocks. The ORM's lazy
`block.edges` makes it look like Python attribute access but hides a database
round-trip. Exploration 017 planned this as a single indexed query per block;
ORM silently makes it worse.

Core forces you to be explicit:
```python
# One query, returns all (block_id, centrality) pairs
stmt = (
    select(edges.c.from_id, func.sum(edges.c.weight).label("centrality"))
    .group_by(edges.c.from_id)
    .union_all(
        select(edges.c.to_id, func.sum(edges.c.weight))
        .group_by(edges.c.to_id)
    )
)
```

**2. Embedding BLOBs**

ORM would represent `blocks.embedding` as a Python `bytes` attribute. Every
time the ORM loads a block object, it deserialises 6KB of bytes. During the
ATTENTION frame scoring pass, all embeddings are loaded — fine. But the SELF
frame scoring pass doesn't need embeddings at all. ORM loads everything in the
mapped object regardless.

Core lets you select only the columns you need:
```python
# SELF frame: no embeddings loaded
stmt = select(
    blocks.c.id, blocks.c.confidence, blocks.c.reinforcement_count,
    blocks.c.decay_lambda, blocks.c.hours_since_reinforcement
).where(...)

# ATTENTION frame: embeddings included
stmt = select(blocks).where(...)
```

**3. Bulk updates**

Reinforcement on every recall() is a bulk UPDATE by primary key set. ORM
requires loading objects, modifying them, and flushing:

```python
# ORM — wasteful: loads 5 objects to update 5 values
for b in session.query(Block).filter(Block.id.in_(ids)).all():
    b.reinforcement_count += 1
    b.hours_since_reinforcement = 0
session.commit()
```

Core executes a single UPDATE statement:

```python
# Core — one UPDATE
conn.execute(
    update(blocks)
    .where(blocks.c.id.in_(ids))
    .values(reinforcement_count=blocks.c.reinforcement_count + 1,
            hours_since_reinforcement=0.0)
)
```

**Verdict: SQLAlchemy Core throughout.** Table definitions give Alembic what it
needs for autogenerated migrations. Explicit queries prevent hidden performance traps.

## Engine Setup: `engine.py`

```python
from sqlalchemy import create_engine, event, text
from sqlalchemy.pool import NullPool

def create_amgs_engine(db_path: str, *, echo: bool = False):
    """
    Create the SQLAlchemy engine for AMGS with correct SQLite settings.
    NullPool: no connection pooling — SQLite is single-file, WAL handles concurrency.
    """
    engine = create_engine(
        f"sqlite:///{db_path}",
        echo=echo,                           # set True during development
        poolclass=NullPool,                  # each context manager gets a fresh connection
        connect_args={"check_same_thread": False},  # required for async-adjacent use
    )

    @event.listens_for(engine, "connect")
    def _set_pragmas(dbapi_conn, _record):
        cursor = dbapi_conn.cursor()
        cursor.execute("PRAGMA journal_mode=WAL")
        cursor.execute("PRAGMA foreign_keys=ON")   # enforce ON DELETE CASCADE
        cursor.execute("PRAGMA synchronous=NORMAL")
        cursor.execute("PRAGMA cache_size=-32000")  # 32 MB page cache
        cursor.execute("PRAGMA temp_store=MEMORY")
        cursor.close()

    return engine


def create_test_engine():
    """In-memory engine for tests. StaticPool ensures one connection = one DB."""
    from sqlalchemy.pool import StaticPool
    engine = create_engine(
        "sqlite:///:memory:",
        connect_args={"check_same_thread": False},
        poolclass=StaticPool,
    )
    from .models import metadata
    metadata.create_all(engine)
    return engine
```

**Why `NullPool`:** SQLite's WAL mode handles concurrent readers natively. A
connection pool for a single-file database adds overhead without benefit and can
cause "database is locked" errors when pool connections hold open transactions.
`NullPool` means each `with engine.connect()` or `engine.begin()` opens and
closes a fresh connection — clean and predictable.

**Why `check_same_thread=False`:** SQLite's default safety check prevents sharing
a connection across threads. `NullPool` + explicit `with` blocks already handle
thread safety correctly; the check is unnecessary overhead.

## Alembic Setup

### `alembic.ini` (key settings)

```ini
[alembic]
script_location = alembic
sqlalchemy.url = sqlite:///%(here)s/amgs.db
```

For the database URL to be dynamic (e.g., different paths per user), override
in `env.py` rather than hardcoding here.

### `alembic/env.py`

```python
from alembic import context
from amgs.db.models import metadata   # import the schema source of truth
from amgs.db.engine import create_amgs_engine

config = context.config

def run_migrations_online():
    engine = create_amgs_engine(config.get_main_option("sqlalchemy.url"))
    with engine.connect() as connection:
        context.configure(
            connection=connection,
            target_metadata=metadata,
            render_as_batch=True,    # ← CRITICAL for SQLite
            compare_type=True,       # detect column type changes
        )
        with context.begin_transaction():
            context.run_migrations()

run_migrations_online()
```

**`render_as_batch=True` is non-negotiable for SQLite.** SQLite has a restricted
`ALTER TABLE` — it cannot add NOT NULL columns without defaults, cannot drop
columns in older versions, cannot add constraints after creation. Alembic's batch
mode works around this by:

1. Creating a new table with the target schema
2. Copying data from the old table
3. Dropping the old table
4. Renaming the new table

This is transparent to the developer but essential for SQLite migrations to work
correctly. Setting it globally in `env.py` means every migration gets it automatically.

**Seed data belongs in the initial migration.** The built-in frames and default
`system_config` values are part of the schema's initial state. If they live in
application code instead, a fresh database without a running application is unusable.

### Phase 2 migration example: `002_add_centrality_cached.py`

```python
"""Add centrality_cached column for Phase 2 materialised centrality

Revision ID: 002
Down revision: 001
"""
from alembic import op
import sqlalchemy as sa

def upgrade():
    with op.batch_alter_table("blocks") as batch_op:   # ← batch mode for SQLite
        batch_op.add_column(
            sa.Column("centrality_cached", sa.Real, nullable=True)
        )
    op.create_index("idx_blocks_centrality", "blocks", ["centrality_cached"])

def downgrade():
    op.drop_index("idx_blocks_centrality", table_name="blocks")
    with op.batch_alter_table("blocks") as batch_op:
        batch_op.drop_column("centrality_cached")
```

All future additive changes follow this pattern — always `batch_alter_table`,
even for `add_column`, because it's safe and consistent.

## Query Layer: Named Functions

Raw SQL strings scattered across business logic make the system hard to test,
refactor, and reason about. All database access goes through named functions.

### `queries/blocks.py`

```python
from sqlalchemy import select, update, and_, func
from ..models import blocks, block_tags

def get_active_blocks_for_scoring(conn):
    """Load scoring fields for all active blocks. No embeddings."""
    stmt = select(
        blocks.c.id,
        blocks.c.confidence,
        blocks.c.reinforcement_count,
        blocks.c.decay_lambda,
        blocks.c.hours_since_reinforcement,
        blocks.c.self_alignment,
    ).where(blocks.c.status == "active")
    return conn.execute(stmt).mappings().all()


def get_active_blocks_with_embeddings(conn):
    """Load all active blocks including embeddings (ATTENTION frame)."""
    stmt = select(blocks).where(
        and_(blocks.c.status == "active", blocks.c.embedding.isnot(None))
    )
    return conn.execute(stmt).mappings().all()


def get_self_tagged_blocks(conn):
    """Active blocks with confirmed self/* tags (SELF frame)."""
    stmt = (
        select(blocks)
        .join(block_tags, blocks.c.id == block_tags.c.block_id)
        .where(
            and_(
                blocks.c.status == "active",
                block_tags.c.tag.like("self/%"),
                block_tags.c.status == "confirmed",
            )
        )
        .distinct()
    )
    return conn.execute(stmt).mappings().all()


def reinforce_blocks(conn, block_ids: list[str], session_id: str) -> None:
    """Reinforce returned blocks after recall(). Call within a transaction."""
    stmt = (
        update(blocks)
        .where(blocks.c.id.in_(block_ids))
        .values(
            reinforcement_count=blocks.c.reinforcement_count + 1,
            hours_since_reinforcement=0.0,
            last_reinforced_session=session_id,
        )
    )
    conn.execute(stmt)


def bulk_apply_session_hours(conn, active_hours: float, before_session_id: str) -> None:
    """Add active_hours to all blocks not reinforced in the current session.
    Called at session start for session-aware decay."""
    stmt = (
        update(blocks)
        .where(
            and_(
                blocks.c.status == "active",
                blocks.c.last_reinforced_session != before_session_id,
            )
        )
        .values(hours_since_reinforcement=blocks.c.hours_since_reinforcement + active_hours)
    )
    conn.execute(stmt)
```

### `queries/recall.py` — the full recall pipeline

```python
from sqlalchemy import select, and_
from ..models import edges, contradictions

def get_block_centrality(conn, block_id: str) -> float:
    """Sum of edge weights for a block (its centrality score)."""
    stmt = select(func.coalesce(func.sum(edges.c.weight), 0.0)).where(
        or_(edges.c.from_id == block_id, edges.c.to_id == block_id)
    )
    return conn.execute(stmt).scalar()


def get_active_contradictions_for_candidates(conn, candidate_ids: list[str]):
    """Check for unresolved contradictions among a set of candidate block IDs."""
    stmt = select(
        contradictions.c.block_a_id,
        contradictions.c.block_b_id,
        contradictions.c.strength,
    ).where(
        and_(
            or_(
                contradictions.c.block_a_id.in_(candidate_ids),
                contradictions.c.block_b_id.in_(candidate_ids),
            ),
            contradictions.c.resolved == 0,
        )
    )
    return conn.execute(stmt).mappings().all()


def reinforce_co_retrieved_edges(conn, block_ids: list[str]) -> None:
    """Reinforce edges between any two blocks that both appear in the returned set."""
    from itertools import combinations
    pairs = [
        (min(a, b), max(a, b))
        for a, b in combinations(block_ids, 2)
    ]
    if not pairs:
        return
    for from_id, to_id in pairs:
        conn.execute(
            update(edges)
            .where(and_(edges.c.from_id == from_id, edges.c.to_id == to_id))
            .values(
                reinforcement_count=edges.c.reinforcement_count + 1,
                hours_since_co_retrieval=0.0,
            )
        )
```

## Transaction Usage Patterns

SQLAlchemy 2.0 uses two context manager patterns:

```python
# Write operations — auto-commit on exit, auto-rollback on exception
with engine.begin() as conn:
    reinforce_blocks(conn, block_ids, session_id)
    reinforce_co_retrieved_edges(conn, block_ids)
# ← committed atomically here

# Read operations — no commit needed
with engine.connect() as conn:
    blocks_data = get_active_blocks_for_scoring(conn)
```

Never mix read and write connections. If a single operation needs both (e.g.,
read then write), use `engine.begin()` for the entire scope.

### The recall() transaction boundary

```python
def recall(frame: str, query: str | None, top_k: int) -> FrameResult:
    # Reads: no transaction needed
    with engine.connect() as conn:
        raw_blocks = get_active_blocks_for_scoring(conn)
        # ... scoring, contradiction suppression, ranking ...
        final_ids = [b.id for b in top_k_blocks]

    # Writes: single atomic transaction
    with engine.begin() as conn:
        reinforce_blocks(conn, final_ids, current_session_id)
        reinforce_co_retrieved_edges(conn, final_ids)

    return FrameResult(...)
```

Read and write are separate connections on purpose. The read takes a snapshot;
the write commits the reinforcement. WAL mode ensures consolidate() writes
don't block the read phase.

## Embedding Handling: numpy ↔ BLOB

Embeddings are stored as `LargeBinary` (BLOB). Convert to/from numpy at the
application boundary — not inside query functions:

```python
import numpy as np

def embedding_to_blob(vec: np.ndarray) -> bytes:
    """Convert numpy float32 array to bytes for SQLite storage."""
    return vec.astype(np.float32).tobytes()

def blob_to_embedding(blob: bytes) -> np.ndarray:
    """Convert stored bytes back to numpy float32 array."""
    return np.frombuffer(blob, dtype=np.float32)

# At consolidation — store:
conn.execute(
    insert(blocks).values(embedding=embedding_to_blob(vec), ...)
)

# At recall() — load and convert all at once:
rows = get_active_blocks_with_embeddings(conn)
embeddings = np.vstack([blob_to_embedding(r["embedding"]) for r in rows])
block_ids = [r["id"] for r in rows]
# then: cosine_similarity(query_vec, embeddings)
```

The conversion functions are the **only** place where numpy↔bytes conversion
happens. Nowhere else in the codebase should `tobytes()` or `frombuffer()` appear.

## Testing Strategy

Every query function can be tested against an in-memory SQLite database:

```python
# conftest.py
import pytest
from amgs.db.engine import create_test_engine

@pytest.fixture
def db():
    engine = create_test_engine()
    yield engine
    engine.dispose()

# test_queries.py
def test_reinforce_blocks(db):
    with db.begin() as conn:
        conn.execute(insert(blocks).values(
            id="abc", file_path="blocks/abc.md", category="knowledge/technical",
            source="test", created_at="2026-01-01T00:00:00Z",
            confidence=0.5, reinforcement_count=0, decay_lambda=0.01,
            hours_since_reinforcement=10.0
        ))

    with db.begin() as conn:
        reinforce_blocks(conn, ["abc"], session_id="sess_001")

    with db.connect() as conn:
        result = conn.execute(select(blocks.c.reinforcement_count, blocks.c.hours_since_reinforcement)
                              .where(blocks.c.id == "abc")).one()
    assert result.reinforcement_count == 1
    assert result.hours_since_reinforcement == 0.0
```

`create_test_engine()` with `StaticPool` ensures all connections in a test use
the same in-memory database. Tests are fast, isolated, and require no file cleanup.

## Running Migrations

```bash
# Initialise Alembic (first time only)
alembic init alembic

# Create a new migration (autogenerate from models.py diff)
alembic revision --autogenerate -m "add centrality_cached"

# Apply migrations to current database
alembic upgrade head

# Rollback one version
alembic downgrade -1

# Show current migration state
alembic current

# Show migration history
alembic history
```

**Always commit migration files to version control.** The `alembic_version` table
in the database records which migration the database is at. A fresh deployment
runs `alembic upgrade head` and arrives at the correct schema.

## Design Decisions from this Exploration

| Decision | Rationale |
|----------|-----------|
| SQLAlchemy Core, not ORM | No N+1 centrality queries; explicit column selection avoids loading embeddings unnecessarily; bulk UPDATE is a single statement |
| `NullPool` for SQLite engine | Avoids "database is locked" from pooled connections holding open transactions |
| `render_as_batch=True` globally in `env.py` | SQLite ALTER TABLE is restricted; batch mode is always safe, never harmful |
| `check_same_thread=False` with `NullPool` | Thread safety handled by explicit `with` blocks; Python's default check is redundant |
| PRAGMA setup via `@event.listens_for(engine, "connect")` | Every connection — pooled or fresh — gets correct WAL/foreign-key settings |
| `models.py` is the schema source of truth | Alembic, application code, and tests all import from one place; no drift |
| Seed data (built-in frames, system_config defaults) in initial migration | Fresh database is usable without a running application |
| Named query functions, no scattered SQL strings | Testable in isolation; refactorable without grepping codebase |
| Embedding conversion (`tobytes`/`frombuffer`) in one dedicated module | Single conversion boundary; never leaks into query or scoring logic |
| `StaticPool` for in-memory test databases | All test connections see the same data; no test isolation issues |
| Integer (0/1) for boolean columns | SQLite has no BOOLEAN type; explicit integers avoid SQLAlchemy's coercion surprises |
| `op.batch_alter_table` in all migrations, even add_column | Defensive consistency; adding a NOT NULL column later requires batch mode |

## Open Questions

- [ ] Should `embedding_to_blob` / `blob_to_embedding` handle model version
      changes? (e.g., if `embedding_model` changes, stored BLOBs from the old
      model are invalid — need a migration or a re-embedding job)
- [ ] Should query functions accept `conn` as a parameter (current design) or
      should there be a repository class that holds the connection? (Current design
      is simpler and more testable; repository class adds indirection for minimal gain)
- [ ] Should `system_config` values be typed? A `Config` dataclass that reads
      all values at session start and provides typed attributes is cleaner than
      `conn.execute("SELECT value FROM system_config WHERE key=?")` in application code.
- [ ] Should Alembic run automatically on `MemorySystem.__init__()`?
      (`alembic upgrade head` on startup ensures the database is always current,
      but adds latency to the first call; alternatively, a `health_check()` method)
